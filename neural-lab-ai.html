01 / The operational problem

Local inference was powerful — but fragmented

Running models locally required remembering dozens of flags, checking hardware in separate tools, managing process IDs by hand, tracking ports, and diagnosing errors from raw terminal output. The intelligence was local. The operating experience was not yet a system.

terminal

Command complexity

Context size, GPU layers, batching, flash attention, templates, ports, and backend flags had to be assembled correctly for every launch.

monitor_heart

Scattered observability

GPU, RAM, disks, open ports, running processes, logs, and endpoint health lived in different commands and windows.

sync_problem

Repeatable failure

Known-good configurations were easy to lose, while port conflicts, model path errors, and VRAM failures repeatedly interrupted experiments.

04 / System architecture

Local by design, layered by responsibility

The frontend never manages operating-system processes directly. It talks to a FastAPI control layer that validates requests, persists state, builds commands, monitors hardware, and manages llama.cpp process groups.

A local operating surface — not a cloud dependency.

05 / Safety & operational control

Execution has guardrails

verified_user

Command allowlist

The integrated terminal accepts 34 diagnostic commands; destructive and arbitrary execution paths are rejected.

lan

Local default boundary

The standard host is 127.0.0.1, keeping the control surface and inference endpoint on the machine unless deliberately changed.

account_tree

Process-group management

Servers launch in dedicated process groups so stop operations clean up child processes instead of leaving orphaned inference jobs.

07 / My role and the AI's role

The code was AI-generated. The direction, requirements, and acceptance decisions were mine.

This distinction matters. I am not presenting the project as proof that I manually wrote every line of Python or React. It is proof that I can define an operational problem, translate needs into requirements, direct an AI-assisted build, evaluate the output, identify failures, and maintain human responsibility for the final result.

01 · DEFINE

Human identifies the operational problem and sets scope.

02 · PLAN

Agent analyzes architecture, risks, files, and validation.

03 · APPROVE

Human approves, adjusts, or rejects the proposed plan.

04 · EXECUTE

Agent implements only the approved scope.

05 · VALIDATE

Human tests the system and accepts evidence, not claims.

Three critical bugs were found only through human testing in the real environment.

A port conflict prevented startup, a context value and start action failed during product validation, and a TypeScript regression produced a blank frontend. Each issue passed back through diagnosis, plan approval, correction, and retesting. Human-in-the-loop was not a label added after development — it was the mechanism that made the product reliable.
